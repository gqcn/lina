## Context

当前系统的导出导入功能存在以下问题：

1. **导出文件命名**：后端生成的文件名和前端下载时的文件名不一致，且缺乏描述性
2. **字典管理功能分散**：字典类型和字典数据各自有独立的导出导入按钮，用户需要分别操作，流程繁琐

## Goals / Non-Goals

**Goals:**
- 统一所有模块导出文件的命名规范，使用描述性名称（如"字典数据导出.xlsx"）
- 将字典管理的导出导入功能合并到字典类型面板，同时支持类型和数据的导出导入
- 简化用户操作流程，减少重复操作

**Non-Goals:**
- 不修改其他模块（用户、岗位、日志等）的导出导入逻辑
- 不修改数据库结构

## Decisions

### 1. 导出文件命名规范

**规则**：导出文件名称格式为 `{功能描述}导出.xlsx`，与业务功能相关而非模块名称。

| 模块 | 当前文件名 | 新文件名 |
|------|-----------|---------|
| 字典类型 | `dict_types.xlsx` / `字典类型.xlsx` | `字典类型导出.xlsx` |
| 字典数据 | `dict_data.xlsx` / `字典数据.xlsx` | `字典数据导出.xlsx` |
| 参数设置 | `configs.xlsx` | `参数设置导出.xlsx` |
| 用户管理 | - | `用户数据导出.xlsx` |
| 岗位管理 | - | `岗位数据导出.xlsx` |
| 操作日志 | - | `操作日志导出.xlsx` |
| 登录日志 | - | `登录日志导出.xlsx` |

### 2. 字典导出导入重构设计

#### 2.1 Excel 文件格式

采用双 Sheet 结构：

| Sheet | 名称 | 内容 |
|-------|------|------|
| Sheet 1 | 字典类型 | 字典类型数据（名称、类型、状态、备注） |
| Sheet 2 | 字典数据 | 字典数据（所属类型、标签、值、排序、样式、状态、备注） |

#### 2.2 导出行为

**场景一：导出全部**
- Sheet 1 包含所有字典类型
- Sheet 2 包含所有字典数据

**场景二：导出选中类型**
- Sheet 1 包含选中的字典类型
- Sheet 2 包含这些类型下的所有字典数据

**文件名**：`字典管理导出.xlsx`

#### 2.3 导入行为

**处理流程**：
1. 解析 Excel 文件，读取两个 Sheet
2. 先导入字典类型，再导入字典数据
3. 校验字典数据的 `dict_type` 字段必须存在于 Sheet 1 或数据库中

**冲突处理**：
- 字典类型：`type` 字段唯一，已存在则跳过并记录失败原因
- 字典数据：`dict_type + value` 组合唯一，已存在则跳过并记录失败原因

**导入结果**：返回类型和数据各自的成功/失败数量及失败详情

#### 2.4 模板下载

提供包含两个 Sheet 的模板文件，两个 Sheet 都包含示例数据和说明行。

### 3. UI 变更

#### 3.1 字典类型面板

- 保留导出/导入按钮
- 更新导出功能为合并导出（类型+数据）
- 更新导入功能为合并导入

#### 3.2 字典数据面板

- 移除导出按钮
- 移除导入按钮
- 保留新增、删除按钮

### 4. API 变更

#### 4.1 新增合并导出接口

```
GET /dict/export
```
参数与原类型导出接口一致，返回包含两个 Sheet 的 Excel 文件。

#### 4.2 新增合并导入接口

```
POST /dict/import
```
请求：multipart/form-data，上传 Excel 文件
响应：
```json
{
  "typeSuccess": 10,
  "typeFail": 2,
  "dataSuccess": 50,
  "dataFail": 5,
  "failList": [...]
}
```

#### 4.3 废弃接口

以下接口标记为废弃，保留向后兼容：
- `GET /dict/type/export`
- `POST /dict/type/import`
- `GET /dict/data/export`
- `POST /dict/data/import`

### 5. 后端实现

#### 5.1 新增文件

- `api/dict/v1/dict_export.go` - 合并导出 API 定义
- `api/dict/v1/dict_import.go` - 合并导入 API 定义
- `internal/controller/dict/dict_v1_export.go` - 合并导出控制器
- `internal/controller/dict/dict_v1_import.go` - 合并导入控制器
- `internal/service/dict/dict_export.go` - 导出服务逻辑
- `internal/service/dict/dict_import.go` - 导入服务逻辑

#### 5.2 修改文件

- 更新各模块导出控制器的文件名生成逻辑
- 更新前端各模块的导出文件名

## Risks / Trade-offs

- **向后兼容**：保留原有 API 接口，避免影响可能的第三方调用
- **导入复杂度**：合并导入需要处理类型和数据之间的依赖关系，逻辑较复杂
- **用户习惯**：用户可能习惯分开操作，需要更新使用文档
